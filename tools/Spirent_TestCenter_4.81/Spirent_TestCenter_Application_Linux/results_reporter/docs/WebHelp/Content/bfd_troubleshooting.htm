BFD Control-Plane Convergence Troubleshooting Guide

In order for the results of a BFD Control-Plane convergence test to be displayed:

  1. The BFD daemon must be able to communicate convergence trigger events to the Control-Plane Convergence daemon.
  2. Those convergence trigger events must be stored in the daemon.
  3. Those events must be sent up to the BLL.
  4. The events must be stored via the Save Results command into the database file.

Possible issues

  • The BFD -> Stop PDU command is not sent to the Primary devices. This is currently the only supported BFD convergence trigger event, and if the event is not triggered via this method, then the BFD daemon will not communicate to the Control-Plane Convergence daemon.
  • The BFD -> Stop PDU command is improperly sent to both the Primary and its peer device. When this happens, it is indeterminate which device will bring down the BFD session. If the peer device brings the BFD session down, the BFD daemon corresponding to the Primary device doesn’t have the opportunity to communicate to the Control-Plane Convergence daemon.
  • The BFD session is brought down by any other method other than BFD -> Stop PDU. In this case, the BFD daemon will not communicate to the Control-Plane Convergence daemon. Examples include, but are not limited to: Break Link, Stop Devices, stopping the controlling routing protocol, Stop BFD, flaps, or timeouts in the BFD session.
  • The Control-Plane Convergence objects, the Control-Plane Convergence commands, and the BFD -> Stop PDU command’s selected devices are not in sync with each other. If they are not in sync, it is possible for the expected convergence trigger events to not be recorded properly.
  • Test results are cleared using the Clear All Results command after the BFD -> Stop PDU command, but before the Save Results command is performed. In this case, the convergence trigger events that would have been written to the results file are instead lost.
