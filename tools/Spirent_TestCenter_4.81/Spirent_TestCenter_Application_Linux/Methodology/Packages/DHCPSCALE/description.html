Overview

The DHCP Session Scale methodology determines the maximum number of DHCP sessions a DUT can bring up for a given connection rate.

In this methodology, Spirent TestCenter emulates multiple DHCP clients which establish sessions with a DHCP server, which can be the DUT or a Spirent TestCenter port. The methodology starts by attempting to establish a minimum number of DHCP sessions at a given rate; if successful, the number is increased until either the attempted number of sessions cannot be achieved, or the configured maximum number of sessions is achieved. For highest achieved number of sessions, bidirectional traffic is sent between the DHCP Server to the DHCP Clients, after the DHCP sessions are established.

This methodology works with IPv4, IPv6, or both. The methodology can also work with single or stacked (QinQ) VLANs.

  • A DHCP Client Port Group, where Spirent TestCenter ports emulating multiple PPPoE Clients

  • The DUT, which can be a Broadband Network Gateway (BNG), or a DHCP Relay Agent

  • A DHCP Server/Traffic Endpoint port group

Test Steps

The steps in this methodology are as follows:

  1. Search Phase:

    • Perform a binary search to find out the maximum number of DHCP sessions supported by DUT for the given session rate.

    • Record the highest session scale.

    • Create a full mesh traffic configuration between the DHCP Clients and Internet hosts emulated by STC on the DHCP Server side of DUT.

    • Send traffic, if configured.

  2. Stability Phase:

    • Repeat the steps below the same number of times as the configured Stability Repeat Count

    • Establish a number of DHCP sessions supported by DUT for the given session rate.

    • Create a full mesh traffic configuration between the DHCP Clients and Internet hosts emulated by STC on the DHCP Server side of DUT.

    • Send traffic, if configured.

  3. The test passes if,

    • the achieved DHCP session scale in both the Search Phase and the Stability Phase are greater than or equal to configured Session Count Threshold, within the configured tolerance, and

    • at least one test (traffic) packets is received on each of the test ports
